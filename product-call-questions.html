Foundational Decision

Q1audit ref #14Both
Headcount ordering model — keep headcount-driven or move to item-quantity?
Currently all items use the same ordering model: you enter a headcount (number of people), and every item's price is calculated as price/person × headcount. You never select a quantity or size per item — the headcount drives everything.

HH MP v2 removes the global headcount input entirely. Each item type handles quantity differently instead: Platters are priced as a whole item (e.g. "$120, serves 10") with size variations as modifiers. Individual items are priced per unit — you pick a quantity ("3 sandwiches"). This decision gates the serving style tab build, the item modal redesign, the cart behaviour, and whether "Add Headcount" stays in the nav or is removed.

Key Structural Notes

  • Current marketplace is menu-centric, not restaurant-centric. Each menu is its own page. A chef can have multiple menus, each with its own page and its own separate order. HH MP v2 is one restaurant page showing all serving styles together.
  • Mixed-format ordering does not exist. A customer cannot currently order platters and individual items in the same order. This is a core feature of HH MP v2.
  • Both menu types use the same headcount ordering model. Large-format and individually packaged menus both price by number of people (e.g. "$15/person, minimum 8 people"). HH MP v2 removes the global headcount model entirely — Platters become stand-alone items priced as a whole with sizing handled via modifiers; Individual Meals use a standard unit quantity.
  • Individually packaged menus are a separate directory category today but are not a structurally different ordering experience — same page layout, same modal, same cart behaviour.
  • Hungry has several business features HH MP v2 does not account for — these should carry forward, not be treated as gaps: SELECT ACCOUNT (corporate billing), Point of Access, Security Check-In, Pay Later (ACH/Check/NET), pre-authorisation model, gratuity selector, weekday rate.
